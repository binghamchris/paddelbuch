# Requirements Document

## Introduction

Add Tinylytics event tracking to the Paddel Buch site to capture user interactions with map markers, popup buttons, filter controls, dashboard tabs, clipboard actions, navigation links, and the language switcher. Tinylytics already handles page-view analytics; this feature extends it to discrete click events using the `data-tinylytics-event` and `data-tinylytics-event-value` HTML attributes. All event names follow the `category.action` naming convention required by the Tinylytics API.

## Glossary

- **Tinylytics_Script**: The external analytics script loaded from `https://tinylytics.app/embed/DWSnjEu6fgk9s2Yu2H4a/min.js` in `_layouts/default.html`
- **Event_Attribute**: The `data-tinylytics-event` HTML attribute that names the event in `category.action` format
- **Value_Attribute**: The `data-tinylytics-event-value` HTML attribute that carries a distinguishing identifier (slug, key, or locale) for the event
- **Spot_Popup**: The Leaflet popup displayed when a non-rejected spot marker is clicked, rendered from `spot-popup.html` or the JS fallback in `layer-control.js`
- **Rejected_Popup**: The Leaflet popup displayed when a rejected (no-entry) spot marker is clicked, rendered from `rejected-popup.html` or the JS fallback in `layer-control.js`
- **Obstacle_Popup**: The Leaflet popup displayed when an obstacle layer is clicked, rendered from `obstacle-popup.html` or the JS fallback in `layer-control.js`
- **Obstacle_Layer**: The Leaflet GeoJSON layer rendered for an obstacle (and its optional portage route) in `layer-control.js`, which opens an Obstacle_Popup on click
- **Event_Popup**: The Leaflet popup displayed when an event notice marker is clicked, rendered from `event-popup.html` or the JS fallback in `layer-control.js`
- **Protected_Area_Layer**: The Leaflet GeoJSON polygon layer rendered for a protected area in `layer-control.js`, which opens a popup on click
- **Protected_Area_Popup**: The Leaflet popup displayed when a Protected_Area_Layer is clicked, generated in `layer-control.js`
- **Filter_Panel**: The Leaflet control rendered by `filter-panel.js` containing spot-type filter checkboxes, layer toggle checkboxes, and a toggle button
- **Dashboard_Switcher**: The tab-button UI rendered by `dashboard-switcher.js` that switches between data-quality dashboards
- **Navigate_Button**: The external link rendered by `navigate-btn.html` that opens Google Maps directions to a spot
- **Copy_Button**: The clipboard buttons rendered in `spot-detail-content.html` and bound by `clipboard.js` for copying GPS coordinates or addresses
- **Language_Switcher**: The locale dropdown links in `header.html` that switch between German and English

## Requirements

### Requirement 1: Enable Event Collection on the Tinylytics Script

**User Story:** As a site owner, I want the Tinylytics embed script to collect click events, so that interaction data is sent to the Tinylytics dashboard alongside page views.

#### Acceptance Criteria

1. THE Tinylytics_Script `src` URL SHALL include the `?events` query parameter to enable event collection
2. THE Tinylytics_Script `src` URL SHALL include the `beacon` query parameter to use `navigator.sendBeacon` for outbound link events
3. THE Tinylytics_Script SHALL remain loaded with the `defer` attribute to avoid blocking page rendering

### Requirement 2: Track Map Marker and Layer Click Events

**User Story:** As a site owner, I want to know which map markers, obstacles, and protected areas visitors click, so that I can understand which spots, obstacles, protected areas, and event notices attract the most interest.

#### Acceptance Criteria

1. WHEN a non-rejected spot marker is clicked, THE Spot_Popup SHALL carry an Event_Attribute with the value `marker.click` and a Value_Attribute containing the spot slug
2. WHEN a rejected spot marker is clicked, THE Rejected_Popup SHALL carry an Event_Attribute with the value `marker.click` and a Value_Attribute containing the spot slug
3. WHEN an Obstacle_Layer is clicked, THE Obstacle_Popup SHALL carry an Event_Attribute with the value `marker.click` and a Value_Attribute containing the obstacle slug
4. WHEN an event notice marker is clicked, THE Event_Popup SHALL carry an Event_Attribute with the value `marker.click` and a Value_Attribute containing the notice slug
5. WHEN a Protected_Area_Layer is clicked, THE Protected_Area_Popup SHALL carry an Event_Attribute with the value `marker.click` and a Value_Attribute containing the protected area slug or name
6. WHEN popup content is generated by the JavaScript fallback in `layer-control.js`, THE fallback HTML SHALL include the same Event_Attribute and Value_Attribute as the Jekyll include equivalent

### Requirement 3: Track Popup Navigate Button Events

**User Story:** As a site owner, I want to know when visitors request directions from a popup, so that I can measure how often the map drives real-world navigation.

#### Acceptance Criteria

1. WHEN a visitor clicks the "Navigate" button inside a Spot_Popup, THE Navigate button element SHALL carry an Event_Attribute with the value `popup.navigate` and a Value_Attribute containing the spot slug
2. WHEN popup content is generated by the JavaScript fallback in `layer-control.js`, THE fallback Navigate button HTML SHALL include the same Event_Attribute and Value_Attribute as the Jekyll include equivalent

### Requirement 4: Track Popup More-Details Button Events

**User Story:** As a site owner, I want to know when visitors navigate from a popup to a detail page, so that I can measure engagement depth from the map.

#### Acceptance Criteria

1. WHEN a visitor clicks the "More details" button inside a Spot_Popup, THE More-details button element SHALL carry an Event_Attribute with the value `popup.details` and a Value_Attribute containing the spot slug
2. WHEN a visitor clicks the "More details" button inside a Rejected_Popup, THE More-details button element SHALL carry an Event_Attribute with the value `popup.details` and a Value_Attribute containing the spot slug
3. WHEN a visitor clicks the "More details" button inside an Obstacle_Popup, THE More-details button element SHALL carry an Event_Attribute with the value `popup.details` and a Value_Attribute containing the obstacle slug
4. WHEN a visitor clicks the "More details" button inside an Event_Popup, THE More-details button element SHALL carry an Event_Attribute with the value `popup.details` and a Value_Attribute containing the notice slug
5. WHEN popup content is generated by the JavaScript fallback in `layer-control.js`, THE fallback More-details button HTML SHALL include the same Event_Attribute and Value_Attribute as the Jekyll include equivalent

### Requirement 5: Track Filter Panel Toggle Events

**User Story:** As a site owner, I want to know how often visitors open and close the filter panel, so that I can gauge interest in the filtering feature.

#### Acceptance Criteria

1. THE Filter_Panel toggle button SHALL carry an Event_Attribute with the value `filter.toggle`

### Requirement 6: Track Spot-Type Filter Checkbox Events

**User Story:** As a site owner, I want to know which spot-type and paddle-craft filters visitors change, so that I can understand which filter dimensions are most used.

#### Acceptance Criteria

1. WHEN a spot-type filter checkbox is created by the Filter_Panel, THE checkbox element SHALL carry an Event_Attribute with the value `filter.change` and a Value_Attribute in the format `dimension_key:option_slug`

### Requirement 7: Track Layer Toggle Events

**User Story:** As a site owner, I want to know which map layers visitors toggle on or off, so that I can understand which data layers are most valued.

#### Acceptance Criteria

1. WHEN a layer toggle checkbox is created by the Filter_Panel, THE checkbox element SHALL carry an Event_Attribute with the value `layer.toggle` and a Value_Attribute containing the layer key

### Requirement 8: Track Dashboard Switcher Tab Events

**User Story:** As a site owner, I want to know which data-quality dashboard tabs visitors select, so that I can prioritise dashboard development.

#### Acceptance Criteria

1. WHEN a dashboard tab button is created by the Dashboard_Switcher, THE button element SHALL carry an Event_Attribute with the value `dashboard.switch` and a Value_Attribute containing the dashboard id

### Requirement 9: Track Clipboard Copy Events

**User Story:** As a site owner, I want to know when visitors copy GPS coordinates or addresses, so that I can measure how often spot location data is reused.

#### Acceptance Criteria

1. WHEN a Copy_Button for GPS coordinates is present on a spot detail page, THE button element SHALL carry an Event_Attribute with the value `clipboard.copy-gps` and a Value_Attribute containing the spot slug
2. WHEN a Copy_Button for an address is present on a spot detail page, THE button element SHALL carry an Event_Attribute with the value `clipboard.copy-address` and a Value_Attribute containing the spot slug

### Requirement 10: Track Detail-Page Navigate Button Events

**User Story:** As a site owner, I want to know when visitors request directions from a detail page, so that I can compare navigation intent between the map and detail pages.

#### Acceptance Criteria

1. WHEN a Navigate_Button is rendered on a detail page, THE link element SHALL carry an Event_Attribute with the value `navigate.click` and a Value_Attribute containing the spot slug

### Requirement 11: Track Language Switcher Events

**User Story:** As a site owner, I want to know when visitors switch languages, so that I can understand the demand for each locale.

#### Acceptance Criteria

1. WHEN a non-active Language_Switcher link is rendered in the header, THE link element SHALL carry an Event_Attribute with the value `language.switch` and a Value_Attribute containing the target locale code

### Requirement 12: CSP and Attribute Compliance

**User Story:** As a developer, I want all event tracking to comply with the existing Content Security Policy, so that no browser errors or blocked requests occur.

#### Acceptance Criteria

1. THE event tracking implementation SHALL add Event_Attribute and Value_Attribute exclusively as HTML element attributes, without introducing inline scripts or inline style attributes
2. FOR static HTML elements rendered by Jekyll includes, THE Event_Attribute and Value_Attribute SHALL be added directly in the Liquid template markup
3. FOR dynamically generated HTML elements rendered by JavaScript modules, THE Event_Attribute and Value_Attribute SHALL be set using `setAttribute` or string concatenation in the external `.js` file where the element is created
4. THE Content Security Policy defined in `deploy/frontend-deploy.yaml` SHALL remain unchanged
