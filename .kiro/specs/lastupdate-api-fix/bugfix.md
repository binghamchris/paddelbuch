# Bugfix Requirements Document

## Introduction

The JSON API files generated by the Jekyll site do not match the output of the original Gatsby site (`https://www.paddelbuch.ch/api/`). The Jekyll migration introduced structural regressions across all API endpoints: fact tables, dimension tables, and the last update index. The API is consumed by downstream clients that depend on the exact field names, nesting structure, timestamp format, and JSON formatting. All API output must match the Gatsby site exactly.

The issues fall into six categories:
1. File naming differences (2 files have wrong names)
2. Fact table structure differences (field names, nesting, data types)
3. Dimension table structure differences (missing fields, extra fields)
4. Timestamp format differences (milliseconds stripped, wrong timezone suffix)
5. JSON formatting differences (pretty-printed vs compact)
6. Last update index differences (already documented, locale-suffixed entries vs consolidated)

## Bug Analysis

### Current Behavior (Defect)

#### A. File Naming

1.1 WHEN the Jekyll site generates the waterway event notices fact table THEN the system names the files `notices-de.json` / `notices-en.json` instead of the Gatsby-compatible names `waterwayevents-de.json` / `waterwayevents-en.json`

1.2 WHEN the Jekyll site generates the protected areas fact table THEN the system names the files `protected-areas-de.json` / `protected-areas-en.json` (with hyphen) instead of the Gatsby-compatible names `protectedareas-de.json` / `protectedareas-en.json` (no hyphen)

#### B. Fact Table Structure — Spots

1.3 WHEN the Jekyll site generates spots JSON THEN the system uses the field name `locale` instead of the Gatsby field name `node_locale`

1.4 WHEN the Jekyll site generates spots JSON THEN the system places `locale` as the first field instead of `slug` as the first field (Gatsby field order: `slug`, `node_locale`, `createdAt`, `updatedAt`, `name`, ...)

1.5 WHEN the Jekyll site generates the `description` field for spots THEN the system renders it as an HTML string (e.g., `"<p>text</p>"`) instead of preserving the Contentful rich text structure as `{"raw": "{...JSON...}"}`. Empty descriptions output `""` instead of `{"raw": "{\"data\":{},\"content\":[],\"nodeType\":\"document\"}"}`

1.6 WHEN the Jekyll site generates the `approximateAddress` field for spots THEN the system flattens it to a plain string (e.g., `"Flösserstrasse 7, 5000 Aarau"`) instead of preserving the Gatsby nested object structure `{"approximateAddress": "Flösserstrasse 7, 5000 Aarau"}`

1.7 WHEN the Jekyll site generates the `rejected` field for spots THEN the system converts `null` values to `false` instead of preserving `null`

1.8 WHEN the Jekyll site generates reference fields for spots (waterway, spotType, paddlingEnvironmentType, dataSourceType, dataLicenseType) THEN the system flattens them to `_slug` string fields (e.g., `"waterway_slug": "aare"`) instead of preserving the Gatsby nested object structure (e.g., `"waterway": {"slug": "aare"}`)

1.9 WHEN the Jekyll site generates the paddle craft type field for spots THEN the system uses the field name `paddleCraftTypes` with an array of strings (e.g., `["seekajak", "kanadier"]`) instead of the Gatsby field name `paddleCraftType` with an array of objects (e.g., `[{"slug": "seekajak"}, {"slug": "kanadier"}]`)

1.10 WHEN the Jekyll site generates the waterway event notice field for spots THEN the system uses the field name `eventNotices` with an empty array `[]` instead of the Gatsby field name `waterway_event_notice` with `null` (when no notices) or an array of objects with `slug`, `startDate`, `endDate` fields

1.11 WHEN the Jekyll site generates the obstacle field for spots THEN the system uses the field name `obstacles` with an empty array `[]` instead of the Gatsby field name `obstacle` with `null` (when no obstacles) or an array of objects with `slug` field

#### C. Fact Table Structure — Obstacles

1.12 WHEN the Jekyll site generates obstacles JSON THEN the system uses `locale` instead of `node_locale` and places it first instead of `slug`

1.13 WHEN the Jekyll site generates the `description` field for obstacles THEN the system renders it as HTML instead of preserving `{"raw": "{...}"}` (same as spots issue 1.5). Null descriptions output `null` instead of `null` (this is correct for obstacles)

1.14 WHEN the Jekyll site generates the `geometry` field for obstacles THEN the system flattens it to a JSON string instead of preserving the Gatsby nested structure `{"internal": {"content": "{...GeoJSON...}"}}`

1.15 WHEN the Jekyll site generates the `portageRoute` field for obstacles THEN the system flattens it to a JSON string instead of preserving `{"internal": {"content": "{...GeoJSON...}"}}`

1.16 WHEN the Jekyll site generates the `portageDescription` field for obstacles THEN the system renders it as HTML instead of preserving `{"raw": "{...}"}`

1.17 WHEN the Jekyll site generates reference fields for obstacles (obstacleType, waterway, dataSourceType, dataLicenseType) THEN the system flattens them to `_slug` string fields instead of nested objects (same pattern as spots issue 1.8)

1.18 WHEN the Jekyll site generates obstacles JSON THEN the system includes a `spots` field (array of slug strings) that does not exist in the Gatsby output. The Gatsby obstacles query does not include a `spots` field.

#### D. Fact Table Structure — Waterway Events (Notices)

1.19 WHEN the Jekyll site generates waterway event notices JSON THEN the system uses `locale` instead of `node_locale` and places it first instead of `slug`

1.20 WHEN the Jekyll site generates the `description` field for notices THEN the system renders it as HTML instead of preserving `{"raw": "{...}"}`

1.21 WHEN the Jekyll site generates the `affectedArea` field for notices THEN the system flattens it to a JSON string instead of preserving `{"internal": {"content": "{...GeoJSON...}"}}`

1.22 WHEN the Jekyll site generates the `startDate` and `endDate` fields for notices THEN the system outputs full ISO 8601 datetime strings (e.g., `"2020-07-01T00:00:00+00:00"`) instead of date-only strings (e.g., `"2020-07-01"`)

1.23 WHEN the Jekyll site generates the waterway reference for notices THEN the system uses `waterways` (array of slug strings) instead of the Gatsby field name `waterway` (array of objects with `slug` field)

1.24 WHEN the Jekyll site generates the spot reference for notices THEN the system omits the `spot` field entirely instead of including `"spot": null` or `"spot": [{"slug": "..."}]` as in the Gatsby output

1.25 WHEN the Jekyll site generates notices JSON THEN the system includes a `location` field that does not exist in the Gatsby waterway events output

1.26 WHEN the Jekyll site generates reference fields for notices (dataSourceType, dataLicenseType) THEN the system omits these fields instead of including them as nested objects

#### E. Fact Table Structure — Waterways

1.27 WHEN the Jekyll site generates waterways JSON THEN the system uses `locale` instead of `node_locale` and places it first instead of `slug`

1.28 WHEN the Jekyll site generates the `geometry` field for waterways THEN the system flattens it to a JSON string instead of preserving `{"internal": {"content": "{...GeoJSON...}"}}`

1.29 WHEN the Jekyll site generates reference fields for waterways (paddlingEnvironmentType, dataSourceType, dataLicenseType) THEN the system flattens them to `_slug` string fields instead of nested objects

1.30 WHEN the Jekyll site generates waterways JSON THEN the system includes a `showInMenu` field that does not exist in the Gatsby waterways output

#### F. Fact Table Structure — Protected Areas

1.31 WHEN the Jekyll site generates protected areas JSON THEN the system uses `locale` instead of `node_locale` and places it first instead of `slug`

1.32 WHEN the Jekyll site generates the `geometry` field for protected areas THEN the system flattens it to a JSON string instead of preserving `{"internal": {"content": "{...GeoJSON...}"}}`

1.33 WHEN the Jekyll site generates protected areas JSON THEN the system omits the `description` field that exists in the Gatsby output (as `null` or `{"raw": "{...}"}`)

1.34 WHEN the Jekyll site generates the `protectedAreaType` reference for protected areas THEN the system uses `protectedAreaType_slug` (string) instead of `protectedAreaType` (object with `slug`)

1.35 WHEN the Jekyll site generates protected areas JSON THEN the system omits the `waterway` field that exists in the Gatsby output as an array of objects with `slug`

1.36 WHEN the Jekyll site generates protected areas JSON THEN the system omits the `dataSourceType` and `dataLicenseType` fields that exist in the Gatsby output as nested objects

#### G. Dimension Table Structure

1.37 WHEN the Jekyll site generates dimension table JSON (spottypes, obstacletypes, paddlecrafttypes, paddlingenvironmenttypes, protectedareatypes, datasourcetypes, datalicensetypes) THEN the system omits the `node_locale` field that exists in the Gatsby output

1.38 WHEN the Jekyll site generates dimension table JSON THEN the field order is `slug`, `name`, `createdAt`, `updatedAt` instead of the Gatsby order `slug`, `node_locale`, `createdAt`, `updatedAt`, `name`

1.39 WHEN the Jekyll site generates the `paddlecrafttypes` dimension table THEN the system omits the `description` field (as `{"raw": "{...}"}`) that exists in the Gatsby output

1.40 WHEN the Jekyll site generates the `datasourcetypes` dimension table THEN the system omits the `description` field (as `{"raw": "{...}"}`) that exists in the Gatsby output

1.41 WHEN the Jekyll site generates the `datalicensetypes` dimension table THEN the system omits the `summaryUrl` and `fullTextUrl` fields that exist in the Gatsby output

#### H. Timestamp Format (All Tables)

1.42 WHEN the Jekyll site generates `createdAt` and `updatedAt` timestamps in any API JSON file THEN the system truncates them to second precision with `+00:00` suffix (e.g., `"2023-11-23T09:28:19+00:00"`) instead of preserving the Contentful millisecond precision with `Z` suffix (e.g., `"2023-11-23T09:28:19.345Z"`)

#### I. JSON Formatting (All Tables)

1.43 WHEN the Jekyll site generates any API JSON file THEN the system outputs pretty-printed JSON with indentation and newlines instead of compact single-line JSON as produced by the Gatsby site

#### J. Last Update Index

1.44 WHEN the Jekyll site generates `lastUpdateIndex.json` THEN the system produces 24 entries (one per table-locale combination) instead of 12 entries (one per content type)

1.45 WHEN the Jekyll site generates table names for `lastUpdateIndex.json` THEN the system uses lowercase-hyphenated names with locale suffixes (e.g., `spots-de`, `notices-en`, `protected-areas-de`) instead of camelCase Contentful content type names with an `s` suffix (e.g., `spots`, `waterwayEventNotices`, `protectedAreas`)

1.46 WHEN the Jekyll site generates `lastUpdatedAt` timestamps for a content type in `lastUpdateIndex.json` THEN the system reports separate timestamps per locale instead of a single timestamp representing the most recent update across all locales

### Expected Behavior (Correct)

#### File Naming
2.1 The waterway event notices files SHALL be named `waterwayevents-de.json` / `waterwayevents-en.json`
2.2 The protected areas files SHALL be named `protectedareas-de.json` / `protectedareas-en.json` (no hyphen)

#### Fact Tables — General
2.3 All fact table entries SHALL use `node_locale` instead of `locale` as the locale field name
2.4 All fact table entries SHALL place `slug` as the first field, followed by `node_locale`, `createdAt`, `updatedAt`, `name`, then remaining fields
2.5 All `description` fields SHALL preserve the Contentful rich text structure as `{"raw": "{...JSON...}"}` instead of rendering to HTML. Empty descriptions SHALL output `{"raw": "{\"data\":{},\"content\":[],\"nodeType\":\"document\"}"}`
2.6 All `geometry` fields SHALL preserve the nested structure `{"internal": {"content": "{...GeoJSON...}"}}` instead of flattening to a string
2.7 All reference fields to single entities SHALL use nested object format `{"slug": "value"}` instead of `_slug` string format
2.8 All reference fields to arrays of entities SHALL use array of objects format `[{"slug": "value"}]` instead of array of strings

#### Fact Tables — Spots
2.9 The `approximateAddress` field SHALL preserve the nested object structure `{"approximateAddress": "value"}`
2.10 The `rejected` field SHALL preserve `null` when the Contentful value is null, not convert to `false`
2.11 The paddle craft type field SHALL be named `paddleCraftType` (not `paddleCraftTypes`) and contain an array of objects
2.12 The waterway event notice field SHALL be named `waterway_event_notice` (not `eventNotices`) and output `null` when empty, or an array of objects with `slug`, `startDate`, `endDate`
2.13 The obstacle field SHALL be named `obstacle` (not `obstacles`) and output `null` when empty, or an array of objects with `slug`

#### Fact Tables — Obstacles
2.14 The `portageRoute` field SHALL preserve `{"internal": {"content": "{...}"}}` structure
2.15 The `portageDescription` field SHALL preserve `{"raw": "{...}"}` structure
2.16 The obstacles output SHALL NOT include a `spots` field (not present in Gatsby output)

#### Fact Tables — Waterway Events
2.17 The `startDate` and `endDate` fields SHALL output date-only strings (e.g., `"2020-07-01"`) not full datetime strings
2.18 The waterway reference SHALL be named `waterway` and contain an array of objects with `slug`
2.19 The spot reference SHALL be included as `"spot": null` or `"spot": [{"slug": "..."}]`
2.20 The output SHALL include `dataSourceType` and `dataLicenseType` as nested objects
2.21 The output SHALL NOT include a `location` field (not present in Gatsby output)
2.22 The `affectedArea` field SHALL preserve `{"internal": {"content": "{...}"}}` structure

#### Fact Tables — Waterways
2.23 The output SHALL NOT include a `showInMenu` field (not present in Gatsby output)

#### Fact Tables — Protected Areas
2.24 The output SHALL include a `description` field (as `null` or `{"raw": "{...}"}`)
2.25 The output SHALL include a `waterway` field as an array of objects with `slug`
2.26 The output SHALL include `dataSourceType` and `dataLicenseType` as nested objects

#### Dimension Tables
2.27 All dimension table entries SHALL include the `node_locale` field
2.28 All dimension table entries SHALL use field order: `slug`, `node_locale`, `createdAt`, `updatedAt`, `name`, then any additional fields
2.29 The `paddlecrafttypes` dimension table SHALL include the `description` field as `{"raw": "{...}"}`
2.30 The `datasourcetypes` dimension table SHALL include the `description` field as `{"raw": "{...}"}`
2.31 The `datalicensetypes` dimension table SHALL include the `summaryUrl` and `fullTextUrl` fields

#### Timestamps
2.32 All `createdAt` and `updatedAt` timestamps SHALL preserve Contentful millisecond precision with `Z` suffix (e.g., `"2023-11-23T09:28:19.345Z"`)

#### JSON Formatting
2.33 All API JSON files SHALL be output as compact single-line JSON (no pretty-printing)

#### Last Update Index
2.34 `lastUpdateIndex.json` SHALL produce exactly 12 entries, one per content type
2.35 Table names SHALL use camelCase Contentful content type ID with `s` suffix: `spots`, `obstacles`, `waterwayEventNotices`, `protectedAreas`, `waterways`, `dataLicenseTypes`, `dataSourceTypes`, `obstacleTypes`, `paddleCraftTypes`, `protectedAreaTypes`, `spotTypes`, `paddlingEnvironmentTypes`
2.36 Each entry SHALL report a single `lastUpdatedAt` timestamp that is the maximum across all locales
2.37 `lastUpdateIndex.json` SHALL be compact single-line JSON (not pretty-printed)

### Unchanged Behavior (Regression Prevention)

3.1 WHEN the Jekyll site generates per-locale fact table and dimension table JSON files THEN the system SHALL CONTINUE TO produce separate locale-specific files

3.2 WHEN the Jekyll site exposes `site.data['last_updates']` for use in Liquid templates THEN the system SHALL CONTINUE TO provide last update data accessible to templates

3.3 WHEN the Jekyll site generates tile data for spatial queries THEN the tile generation SHALL NOT be affected by API structure changes

3.4 WHEN the Jekyll site renders HTML pages using spot, obstacle, waterway, and other data THEN the HTML page rendering SHALL NOT be affected by API structure changes (the HTML templates use the YAML data files, not the API JSON output)
