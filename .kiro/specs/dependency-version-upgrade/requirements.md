# Requirements Document

## Introduction

This feature covers upgrading all runtime and development dependencies in the Paddelbuch Jekyll project to their latest stable versions. The project currently runs Ruby 3.4.1, Node 18, Jekyll 4.4.1, and various Ruby gems and npm packages. The upgrade encompasses Ruby itself, Node.js, all Ruby gems, all npm packages, and any code changes required for compatibility with the new versions. The CI/CD pipeline (AWS Amplify) and local development tooling must also be updated to reflect the new versions.

## Glossary

- **Build_Pipeline**: The AWS Amplify build configuration defined in `amplify.yml` that installs dependencies and builds the Jekyll site
- **Gemfile**: The Ruby dependency manifest that declares gem requirements and version constraints
- **Gemfile_Lock**: The lockfile that pins exact gem versions for reproducible builds
- **Package_JSON**: The Node.js dependency manifest that declares npm package requirements
- **Package_Lock**: The lockfile that pins exact npm package versions for reproducible builds
- **Ruby_Version_File**: The `.ruby-version` file that specifies the project's required Ruby version
- **Jekyll_Site**: The static site generated by Jekyll from source files, plugins, and configuration
- **Plugin_Suite**: The collection of custom Jekyll plugins in the `_plugins/` directory
- **RSpec_Suite**: The Ruby test suite in the `spec/` directory using RSpec and Rantly
- **Jest_Suite**: The JavaScript test suite in the `_tests/` directory using Jest and fast-check
- **Dependency_Manifest**: A collective term for Gemfile, Gemfile.lock, package.json, and package-lock.json
- **Pre_Upgrade_Baseline**: A snapshot of the fully built `_site` directory captured before any dependency upgrades, used as the reference for verifying site integrity after upgrades

## Requirements

### Requirement 1: Site Integrity After Upgrade

**User Story:** As a site owner, I want the site to remain completely unchanged in content, design, and functionality after all dependency upgrades, so that end users experience no differences whatsoever.

#### Acceptance Criteria

1. THE Jekyll_Site SHALL produce page-for-page identical HTML content (all pages, posts, collection items, API files, and sitemaps) when compared against the Pre_Upgrade_Baseline
2. THE Jekyll_Site SHALL produce identical CSS output, preserving all styling, layout, spacing, and responsive behavior when compared against the Pre_Upgrade_Baseline
3. THE Jekyll_Site SHALL produce identical JavaScript behavior, preserving navigation, map rendering, interactive elements, and all client-side functionality when compared against the Pre_Upgrade_Baseline
4. THE Jekyll_Site SHALL include all images, fonts, and static assets present in the Pre_Upgrade_Baseline without modification to file content or file paths
5. WHEN a Pre_Upgrade_Baseline of the `_site` directory is captured before upgrades begin, THE Jekyll_Site built after all upgrades SHALL match the Pre_Upgrade_Baseline in directory structure and file contents
6. IF a dependency upgrade causes a difference between the post-upgrade build output and the Pre_Upgrade_Baseline, THEN THE Jekyll_Site source code SHALL be adjusted to eliminate the difference
7. THE Jekyll_Site SHALL render all pages with the same visual layout and component behavior across desktop and mobile viewports as the Pre_Upgrade_Baseline

### Requirement 2: Upgrade Ruby Version

**User Story:** As a developer, I want to upgrade Ruby to the latest stable version, so that the project benefits from the latest performance improvements, security patches, and language features.

#### Acceptance Criteria

1. THE Ruby_Version_File SHALL specify the latest stable Ruby release version
2. THE Gemfile_Lock SHALL record a BUNDLED WITH version compatible with the latest stable Ruby release
3. THE Build_Pipeline SHALL reference the same Ruby version as the Ruby_Version_File for installation and activation
4. WHEN the latest stable Ruby version is installed, THE RSpec_Suite SHALL pass all existing tests without modification to test assertions

### Requirement 3: Upgrade Node.js Version

**User Story:** As a developer, I want to upgrade Node.js to the latest LTS version, so that the project uses a supported runtime with current security fixes and features.

#### Acceptance Criteria

1. THE Build_Pipeline SHALL install and use the latest Node.js LTS major version
2. WHEN the latest Node.js LTS version is active, THE Jest_Suite SHALL pass all existing tests without modification to test assertions
3. WHEN the latest Node.js LTS version is active, THE Build_Pipeline SHALL execute `npm ci`, `npm run copy-assets`, `npm run download-fonts`, and `npm test` successfully

### Requirement 4: Upgrade Ruby Gem Dependencies

**User Story:** As a developer, I want to upgrade all Ruby gems to their latest compatible versions, so that the project benefits from bug fixes, security patches, and new features in its dependencies.

#### Acceptance Criteria

1. THE Gemfile SHALL specify version constraints that permit the latest stable releases of each declared gem (jekyll, contentful, dotenv, jekyll-multiple-languages-plugin, jekyll-sass-converter, webrick, bundler-audit, rspec, rantly)
2. THE Gemfile_Lock SHALL record resolved versions that are the latest available for each gem
3. WHEN all gems are upgraded, THE Jekyll_Site SHALL build without errors using `bundle exec jekyll build`
4. WHEN all gems are upgraded, THE RSpec_Suite SHALL pass all existing tests
5. IF a gem introduces a breaking API change, THEN THE Plugin_Suite SHALL be updated to use the new API while preserving existing behavior

### Requirement 5: Upgrade npm Dependencies

**User Story:** As a developer, I want to upgrade all npm packages to their latest compatible versions, so that the project uses current, secure, and well-maintained frontend libraries and dev tools.

#### Acceptance Criteria

1. THE Package_JSON SHALL specify version constraints that permit the latest stable releases of each declared package (bootstrap, leaflet, leaflet.locatecontrol, jest, jest-environment-jsdom, @types/jest, fast-check)
2. THE Package_Lock SHALL record resolved versions that are the latest available for each package
3. WHEN all npm packages are upgraded, THE Jest_Suite SHALL pass all existing tests
4. IF a package introduces a breaking API change, THEN THE Jekyll_Site source code SHALL be updated to use the new API while preserving existing behavior

### Requirement 6: Plugin Compatibility

**User Story:** As a developer, I want all custom Jekyll plugins to remain functional after the dependency upgrades, so that site generation continues to work correctly.

#### Acceptance Criteria

1. WHEN all dependencies are upgraded, THE Plugin_Suite SHALL load without Ruby warnings or deprecation errors during `bundle exec jekyll build`
2. IF a plugin uses a deprecated Ruby API or gem API, THEN THE Plugin_Suite SHALL be updated to use the replacement API

### Requirement 7: Test Suite Compatibility

**User Story:** As a developer, I want both test suites (RSpec and Jest) to pass after all upgrades, so that I have confidence the upgrade has not introduced regressions.

#### Acceptance Criteria

1. WHEN all Ruby dependencies are upgraded, THE RSpec_Suite SHALL pass all tests using `bundle exec rspec`
2. WHEN all npm dependencies are upgraded, THE Jest_Suite SHALL pass all tests using `npm test`
3. IF a test framework upgrade changes assertion syntax or configuration API, THEN THE RSpec_Suite or Jest_Suite SHALL be updated to use the new syntax while testing the same behavior
4. WHEN jest-environment-jsdom is upgraded, THE Jest_Suite SHALL continue to execute DOM-dependent property tests correctly
5. THE RSpec_Suite SHALL maintain at least the same number of test examples after all upgrades as before
6. THE Jest_Suite SHALL maintain at least the same number of test cases after all upgrades as before
7. IF compatibility changes require modifying existing source code, THEN THE modified code SHALL be covered by existing or new tests at the same level of coverage as before the modification

### Requirement 8: Build Pipeline Update

**User Story:** As a developer, I want the AWS Amplify build pipeline to use the upgraded versions, so that CI/CD builds match the local development environment.

#### Acceptance Criteria

1. THE Build_Pipeline SHALL specify the same Ruby version as the Ruby_Version_File in its `rvm install` and `rvm use` commands
2. THE Build_Pipeline SHALL specify the latest Node.js LTS major version in its `nvm install` command
3. WHEN the Build_Pipeline executes, THE Build_Pipeline SHALL complete the preBuild and build phases without errors
4. THE Build_Pipeline SHALL produce a `_site` artifact directory containing the fully built Jekyll_Site
