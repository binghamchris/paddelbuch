# Locale Link Routing Bugfix Design

## Overview

When the site is built for the English locale (`en`), the `jekyll-multiple-languages-plugin` generates pages under `/en/...` and sets `site.lang` to `"en"`. However, all internal links in template includes are constructed with hardcoded German path segments (e.g., `'/einstiegsorte/'`, `'/gewaesser/'`) and piped through `relative_url`, which only prepends `site.baseurl` (empty string). No template accounts for the active locale prefix, so every link on the English site routes back to the German version.

The fix introduces a locale-aware URL prefix variable computed from `site.lang` and `site.default_lang`, then prepends it to every hardcoded internal link across all affected includes. For the default locale (`de`), the prefix is empty (preserving current behavior). For non-default locales (`en`), the prefix is `/{lang}`.

## Glossary

- **Bug_Condition (C)**: The user is browsing a non-default locale (English, `/en/`) and clicks any internal navigation link constructed in a template include
- **Property (P)**: All internal links SHALL include the current locale prefix so users remain on their selected language version
- **Preservation**: All links on the default locale (German) site must continue to work without any locale prefix; language switcher, translated content display, and site build output must remain unchanged
- **`site.lang`**: Variable set by `jekyll-multiple-languages-plugin` indicating the current build locale (`"de"` or `"en"`)
- **`site.default_lang`**: The default language configured in `_config.yml` (`"de"`)
- **`relative_url`**: Jekyll filter that prepends `site.baseurl` to a path; does NOT handle locale prefixes
- **locale prefix**: The URL path segment `/{lang}` prepended for non-default locales (e.g., `/en`); empty string for the default locale

## Bug Details

### Fault Condition

The bug manifests when a user is browsing the English site (`site.lang == "en"`) and clicks any internal link generated by the template includes. Every link is constructed as `'/german-path/' | append: slug | append: '/' | relative_url`, which produces `/german-path/{slug}/` regardless of the active locale. The `relative_url` filter only prepends `site.baseurl` (empty), never the locale prefix.

**Formal Specification:**
```
FUNCTION isBugCondition(input)
  INPUT: input of type { currentLocale: string, defaultLocale: string, linkHref: string }
  OUTPUT: boolean

  RETURN input.currentLocale != input.defaultLocale
         AND input.linkHref does NOT start with "/" + input.currentLocale + "/"
         AND input.linkHref is an internal link (starts with "/")
END FUNCTION
```

### Examples

- User on `/en/` clicks "More details" in spot popup → navigates to `/einstiegsorte/zurich-lake-north/` instead of `/en/einstiegsorte/zurich-lake-north/`
- User on `/en/` clicks waterway link on spot detail page → navigates to `/gewaesser/zurichsee/` instead of `/en/gewaesser/zurichsee/`
- User on `/en/` clicks "More details" in obstacle popup → navigates to `/hindernisse/letten-wehr/` instead of `/en/hindernisse/letten-wehr/`
- User on `/en/` clicks event notice link → navigates to `/gewaesserereignisse/hochwasser-2024/` instead of `/en/gewaesserereignisse/hochwasser-2024/`
- User on `/en/` clicks header nav lake link → navigates to `/gewaesser/zurichsee/` instead of `/en/gewaesser/zurichsee/`
- User on `/en/` clicks portage exit spot link → navigates to `/einstiegsorte/exit-spot/` instead of `/en/einstiegsorte/exit-spot/`
- User on `/en/` clicks static page link in header → navigates to `/offene-daten/api-docs/` instead of `/en/offene-daten/api-docs/`
- User on default locale (German, no prefix) clicks any link → correctly navigates to `/einstiegsorte/{slug}/` (no bug, this must be preserved)

## Expected Behavior

### Preservation Requirements

**Unchanged Behaviors:**
- All links on the German (default locale) site must continue to work without any locale prefix in the URL
- The language switcher in the navbar must continue to switch between German and English correctly
- German translated content (labels, navigation text, dates) must continue to display correctly
- English translated content must continue to display correctly
- Jekyll must continue to generate both German (root) and English (`/en/`) versions of all pages
- The navbar brand link must continue to navigate to the locale-appropriate homepage
- Mouse/touch interactions with all UI elements must remain unchanged
- Map popups, copy buttons, navigation buttons, and all non-link functionality must remain unchanged

**Scope:**
All inputs where `site.lang == site.default_lang` (German locale) should be completely unaffected by this fix. The fix only changes how the URL path is constructed when `site.lang != site.default_lang`.

## Hypothesized Root Cause

Based on the code analysis, the root cause is clear and singular:

1. **Missing locale prefix in URL construction**: Every template include constructs internal links by hardcoding the German path segment and piping through `relative_url`. For example in `spot-popup.html`:
   ```liquid
   href="{{ '/einstiegsorte/' | append: spot.slug | append: '/' | relative_url }}"
   ```
   The `relative_url` filter only prepends `site.baseurl` (which is `""`), not the locale prefix. The `jekyll-multiple-languages-plugin` sets `site.lang` per build but provides no automatic URL rewriting for links constructed this way.

2. **No locale-aware URL helper exists**: The codebase has no shared variable, filter, or include that computes a locale prefix from `site.lang` and `site.default_lang`. Each template would need to independently compute this, but none do.

3. **Affected files** (all in `_includes/`):
   - `spot-popup.html` — "More details" link (`/einstiegsorte/`)
   - `obstacle-popup.html` — "More details" link (`/hindernisse/`)
   - `event-popup.html` — "More details" link (`/gewaesserereignisse/`)
   - `header.html` — Lakes, Rivers, Open Data, About dropdown links (`/gewaesser/`, `/offene-daten/`, `/ueber/`, etc.)
   - `spot-detail-content.html` — Waterway link (`/gewaesser/`)
   - `obstacle-detail-content.html` — Waterway link (`/gewaesser/`), exit spot link (`/einstiegsorte/`), re-entry spot link (`/einstiegsorte/`)
   - `notice-detail-content.html` — Waterway links (`/gewaesser/`)
   - `event-list.html` — Event notice links (`/gewaesserereignisse/`)

## Correctness Properties

Property 1: Fault Condition - Internal Links Include Locale Prefix for Non-Default Locale

_For any_ internal link rendered when `site.lang != site.default_lang` (e.g., English locale), the generated `href` SHALL start with `/{lang}/` (e.g., `/en/einstiegsorte/{slug}/`), ensuring the user remains on their selected language version.

**Validates: Requirements 2.1, 2.2, 2.3, 2.4, 2.5, 2.6, 2.7**

Property 2: Preservation - Internal Links Have No Locale Prefix for Default Locale

_For any_ internal link rendered when `site.lang == site.default_lang` (German locale), the generated `href` SHALL NOT include any locale prefix (e.g., `/einstiegsorte/{slug}/`, not `/de/einstiegsorte/{slug}/`), preserving the existing German URL structure exactly as before.

**Validates: Requirements 2.8, 3.1, 3.2, 3.5, 3.6**

## Fix Implementation

### Changes Required

Assuming our root cause analysis is correct:

**Approach**: Compute a `locale_prefix` variable once per template based on `site.lang` and `site.default_lang`, then prepend it to every hardcoded path. For the default locale, `locale_prefix` is `""` (empty), so existing German URLs are unchanged.

**Locale prefix logic** (Liquid):
```liquid
{% if site.lang != site.default_lang %}
  {% assign locale_prefix = '/' | append: site.lang %}
{% else %}
  {% assign locale_prefix = '' %}
{% endif %}
```

Then replace every link pattern from:
```liquid
href="{{ '/einstiegsorte/' | append: spot.slug | append: '/' | relative_url }}"
```
to:
```liquid
href="{{ locale_prefix | append: '/einstiegsorte/' | append: spot.slug | append: '/' | relative_url }}"
```

**Specific Changes per File:**

1. **`_includes/spot-popup.html`**: Add locale prefix to the "More details" link (`/einstiegsorte/`)

2. **`_includes/obstacle-popup.html`**: Add locale prefix to the "More details" link (`/hindernisse/`)

3. **`_includes/event-popup.html`**: Add locale prefix to the "More details" link (`/gewaesserereignisse/`)

4. **`_includes/header.html`**: Add locale prefix to all dropdown links:
   - Lake links (`/gewaesser/{slug}/`)
   - "More lakes" link (`/gewaesser/seen`)
   - River links (`/gewaesser/{slug}/`)
   - "More rivers" link (`/gewaesser/fluesse`)
   - Open Data static page links (`/{menu_slug}/{slug}/`)
   - Open Data API link (`/offene-daten/api`)
   - About static page links (`/{menu_slug}/{slug}/`)

5. **`_includes/spot-detail-content.html`**: Add locale prefix to the waterway link (`/gewaesser/`)

6. **`_includes/obstacle-detail-content.html`**: Add locale prefix to:
   - Waterway link (`/gewaesser/`)
   - Exit spot link (`/einstiegsorte/`)
   - Re-entry spot link (`/einstiegsorte/`)

7. **`_includes/notice-detail-content.html`**: Add locale prefix to waterway links (`/gewaesser/`)

8. **`_includes/event-list.html`**: Add locale prefix to event notice links (`/gewaesserereignisse/`)

## Testing Strategy

### Validation Approach

The testing strategy follows a two-phase approach: first, surface counterexamples that demonstrate the bug on unfixed code, then verify the fix works correctly and preserves existing behavior.

### Exploratory Fault Condition Checking

**Goal**: Surface counterexamples that demonstrate the bug BEFORE implementing the fix. Confirm that links on the English site lack the `/en/` prefix.

**Test Plan**: Build the site for the English locale and inspect the generated HTML for internal links in each affected include. Verify that `href` attributes do not contain the `/en/` prefix. Run these checks on the UNFIXED code to confirm the root cause.

**Test Cases**:
1. **Spot Popup Link Test**: Build English site, inspect spot popup HTML — "More details" link should be missing `/en/` prefix (will fail on unfixed code)
2. **Obstacle Popup Link Test**: Build English site, inspect obstacle popup HTML — link missing `/en/` prefix (will fail on unfixed code)
3. **Event Popup Link Test**: Build English site, inspect event popup HTML — link missing `/en/` prefix (will fail on unfixed code)
4. **Header Nav Link Test**: Build English site, inspect header dropdown links — all missing `/en/` prefix (will fail on unfixed code)
5. **Detail Page Waterway Link Test**: Build English site, inspect spot/obstacle/notice detail pages — waterway links missing `/en/` prefix (will fail on unfixed code)
6. **Portage Spot Link Test**: Build English site, inspect obstacle detail with portage info — exit/re-entry spot links missing `/en/` prefix (will fail on unfixed code)

**Expected Counterexamples**:
- All internal `href` values on English pages start with `/einstiegsorte/`, `/gewaesser/`, `/hindernisse/`, `/gewaesserereignisse/`, or `/{menu_slug}/` without the `/en/` prefix
- Root cause confirmed: no locale prefix computation exists in any template

### Fix Checking

**Goal**: Verify that for all inputs where the bug condition holds, the fixed templates produce locale-prefixed URLs.

**Pseudocode:**
```
FOR ALL link WHERE isBugCondition(link) DO
  result := renderTemplate_fixed(link)
  ASSERT result.href starts with "/" + site.lang + "/"
END FOR
```

### Preservation Checking

**Goal**: Verify that for all inputs where the bug condition does NOT hold (German locale), the fixed templates produce the same URLs as before.

**Pseudocode:**
```
FOR ALL link WHERE NOT isBugCondition(link) DO
  ASSERT renderTemplate_original(link).href = renderTemplate_fixed(link).href
END FOR
```

**Testing Approach**: Property-based testing is recommended for preservation checking because:
- It can generate many slug/path combinations to verify URL construction
- It catches edge cases like empty slugs, slugs with special characters, or missing data
- It provides strong guarantees that German locale behavior is unchanged

**Test Plan**: Build the site for the German locale on UNFIXED code, capture all internal link `href` values, then build on FIXED code and compare. All German locale links must be identical.

**Test Cases**:
1. **German Spot Popup Preservation**: Verify spot popup "More details" links on German site remain unchanged after fix
2. **German Header Nav Preservation**: Verify all header dropdown links on German site remain unchanged after fix
3. **German Detail Page Preservation**: Verify waterway/spot/obstacle links on German detail pages remain unchanged after fix
4. **Language Switcher Preservation**: Verify language switcher continues to produce correct URLs for both locales

### Unit Tests

- Test locale prefix computation: `site.lang == site.default_lang` → empty prefix; `site.lang != site.default_lang` → `/{lang}` prefix
- Test each template include renders correct `href` for English locale (with `/en/` prefix)
- Test each template include renders correct `href` for German locale (no prefix)
- Test edge cases: missing slug, nil waterway, empty static pages list

### Property-Based Tests

- Generate random slugs and verify English locale links always have `/en/` prefix followed by the correct path segment
- Generate random slugs and verify German locale links never have a locale prefix
- Generate random locale/slug combinations and verify the locale prefix logic is consistent across all template includes

### Integration Tests

- Build full site for both locales and verify all internal links on English pages include `/en/` prefix
- Build full site and verify all internal links on German pages have no locale prefix
- Navigate between pages on the English site and verify locale is preserved throughout
- Use language switcher and verify correct locale switching behavior is maintained
