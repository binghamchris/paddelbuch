# Spot tips feature merged into quality-hardening branch

This merge brings the entire spot-tips feature (composite marker icons with modifier overlays, a `spotTipType` filter dimension, tip banners on detail pages, and a new Contentful content type) into the quality-and-tooling-hardening branch. Four merge conflicts were resolved: `contentful_fetcher.rb`, `precompute_generator.rb`, `layer-control.js`, and `docs/testing.md`. The spot-tips code was uprated to comply with quality-hardening conventions (coordinate validation via `isFiniteCoordinate`, no `console.log`, ASCII-only source, SCSS `@use` pattern, escaped-title-only popup fallback, `bindMarkerRecenter`, Tinylytics beacon dispatch). All 904 RSpec and 1037 Jest tests pass.

**Watch for:** Tip banner `description` field renders raw HTML from Contentful rich text without Liquid escaping (confirmed). The `__no_tips__` synthetic filter option relies on a convention string that has no single-source constant (likely). `createCompositeIcon` builds DOM from string concatenation with no URL sanitization on `iconUrl` values sourced from `TIP_MODIFIER_CONFIG` (possible — currently safe because config is hardcoded, but fragile if config becomes data-driven).

## High-level view

The data pipeline extends naturally: `ContentfulFetcher` gains a `spotTipType` entry in `CONTENT_TYPES` and passes `content_type` through to `flatten_entry` so `map_type` can branch on it, extracting bilingual descriptions. The `map_spot` mapper adds `spotTipType_slugs` via `extract_reference_slugs`, making tip associations available downstream without schema changes.

`PrecomputeGenerator` adds the `spotTipType` dimension to `dimensionConfigs` with a synthetic `__no_tips__` option. The filter engine match function implements the correct inverse semantics: spots with no tips match only `__no_tips__`, spots with tips match any selected slug via OR.

`layer-control.js` is the densest conflict resolution. It combines the quality-hardening helpers with the new `createCompositeIcon` function and `tipSlugs` metadata propagation. The composite icon is only applied to non-rejected spots that have tip slugs, falling back to the standard icon otherwise.

Detail-page rendering uses `CollectionGenerator` to pre-resolve tip types into `page.spot_tip_types`, then the `spot-tip-banners.html` include iterates them with Bootstrap alert markup and per-type colour classes. The description field is rendered unescaped (`{{ tip.description }}`), which is the primary concern.

## Tip banner rendering and trust boundary

`spot-tip-banners.html` renders `{{ tip.description }}` — Liquid outputs the string verbatim as HTML. The value is produced by `extract_rich_text_html` in `contentful_mappers.rb`, which applies `html_escape` to text nodes and validates URIs via `safe_uri?`. The HTML is generated safely from Contentful's rich text AST today. The concern is that Liquid provides no second defence: if the mapper's sanitization is ever bypassed (a raw HTML passthrough, a new node type handler that forgets to escape), the template won't catch it.

## `__no_tips__` filter convention

The match function in `map-data-init.js`:

```javascript
spotTipType: function(meta, selected) {
  var tipSlugs = meta.spotTipType_slugs || [];
  if (tipSlugs.length === 0) {
    return selected.has('__no_tips__');
  }
  for (var i = 0; i < tipSlugs.length; i++) {
    if (selected.has(tipSlugs[i])) return true;
  }
  return false;
}
```

The `__no_tips__` string also appears in `precompute_generator.rb` (where the option is added with a hardcoded label per locale: "Spots without tips" / "Einstiegsorte ohne Tipps") and in property tests. This is the only dimension option whose label is not data-driven from Contentful, and the only one using a magic string convention across Ruby and JavaScript without a shared definition.

## SCSS coupling and accessibility

Adding a new tip type in Contentful requires a corresponding `.alert-spot-tip-<slug>` SCSS class and SVG assets in both `assets/images/markers/` and `assets/images/tips/`. This coupling isn't enforced by code — a missing class degrades gracefully (no custom colour, falls back to default alert styling) but a missing SVG produces a broken image.

Each banner uses `role="alert"`, which in ARIA semantics means "important, time-sensitive information." Screen readers will announce each banner immediately on page load. For a spot with 2-3 tips this produces a burst of announcements before the user has oriented themselves on the page.

## Test coverage gaps

Not tested: the end-to-end path from Contentful entry through `flatten_entry` with `content_type = 'spotTipType'` to the YAML file. The specs use `FakeEntry` objects and test the mapper in isolation. Integration with the real Contentful SDK response shape (especially rich text field structure) is untested. This is consistent with how other content types are tested in this codebase, but means a Contentful SDK upgrade could break `spotTipType` without test signal.
